1. OBJETIVO
================================================================================

Predecir cada 1 segundo la magnitud y direccion del movimiento de BTCUSDT
dentro de 15 segundos (5 clases: STRONG_DOWN, DOWN, FLAT, UP, STRONG_UP).

  Input:  estado actual del mercado + ultimos 120 segundos de historia
  Output: 5 probabilidades: P(STRONG_DOWN), P(DOWN), P(FLAT), P(UP), P(STRONG_UP)

  Si P(STRONG_UP) > threshold   -> long con confianza
  Si P(STRONG_DOWN) > threshold -> short con confianza
  Si P(FLAT) dominante          -> no operar

El modelo se ejecuta cada 1 segundo en vivo.
El dataset de entrenamiento se genera con 1 sample por segundo.

  1 dia  =  86,400 samples
  30 dias = 2,592,000 samples
  90 dias = 7,776,000 samples

================================================================================
2. TARGET (lo que predecimos)
================================================================================

En cada instante T:

  price_now    = midprice(T)     = (best_bid + best_ask) / 2
  price_future = midprice(T+15s)

  change_bps = (price_future - price_now) / price_now * 10000

  THRESHOLDS (configurables):
    T1 = 1.5 bps   (~$13 a precio de 87k)  umbral de ruido
    T2 = 5.0 bps   (~$44 a precio de 87k)  umbral de movimiento fuerte

  5 CLASES ORDINALES:

    clase 0: STRONG_DOWN   change_bps < -T2           (caida fuerte)
    clase 1: DOWN          change_bps ∈ [-T2, -T1)    (caida moderada)
    clase 2: FLAT          change_bps ∈ [-T1, +T1]    (sin movimiento real)
    clase 3: UP            change_bps ∈ (+T1, +T2]    (suba moderada)
    clase 4: STRONG_UP     change_bps > +T2            (suba fuerte)

Usamos midprice (no last trade) porque es mas estable y refleja el
consenso del mercado en ese instante exacto.

Por que 5 clases y no binario:
  - Un movimiento de +1.6 bps y uno de +20 bps NO son lo mismo.
    Con binario ambos son target=1 y el modelo los trata igual.
    Para operar, +1.6 bps ni cubre comisiones; +20 bps es ganancia real.

  - La clase FLAT es critica. Si no la entrenamos, el modelo nunca
    aprende a decir "esto es ruido, no operar". Cuando el mercado esta
    quieto, forzaria una prediccion UP o DOWN y generaria falsos
    positivos que cuestan dinero.

  - Las clases STRONG dan conviccion para operar con mas tamano o
    confianza. Las clases UP/DOWN moderadas son señales mas debiles
    que podemos filtrar o usar con menos exposicion.

  Los thresholds T1=1.5 y T2=5.0 bps son puntos de partida. Se deben
  calibrar empiricamente mirando la distribucion real de cambios a 15s.
  Objetivo orientativo de distribucion:
    STRONG_DOWN: ~10-15%
    DOWN:        ~15-20%
    FLAT:        ~30-40%
    UP:          ~15-20%
    STRONG_UP:   ~10-15%

  En inferencia live, el modelo devuelve 5 probabilidades que suman 1:
    P(STRONG_DOWN), P(DOWN), P(FLAT), P(UP), P(STRONG_UP)

  Decisiones de trading:
    - P(STRONG_UP) > threshold   → long con confianza
    - P(STRONG_DOWN) > threshold → short con confianza
    - P(FLAT) alto               → no operar
    - Combinaciones: P(UP) + P(STRONG_UP) vs P(DOWN) + P(STRONG_DOWN)
      para señal direccional ponderada

3. PRINCIPIO FUNDAMENTAL: TODO RELATIVO
================================================================================

Los modelos de arboles (como LightGBM) usan thresholds fijos en los
splits. Si un split aprendio "price > 87000 -> UP", cuando BTC valga
120k ese split es basura.

REGLA: ningun feature contiene un precio, volumen, o cantidad absoluta.
TODO es: porcentajes, ratios, z-scores, bps, o flags binarios.

Ejemplos:
  MAL:  price = 87,451              BIEN: return_5s = +0.012%
  MAL:  spread = $1.0               BIEN: spread_bps = 1.14
  MAL:  signed_vol = 3.8 BTC        BIEN: signed_vol_zscore = +2.1 sigma
  MAL:  bid_depth = 45 BTC          BIEN: bid_depth_ratio = 0.60
  MAL:  open_interest = 77,329 BTC  BIEN: oi_change_pct = +0.15%

Para volumenes y cantidades que dependen del regimen de mercado, se
normalizan contra la actividad reciente (z-score sobre ventana de 60s
o ratio contra el total).

5. FEATURES - DETALLE COMPLETO
================================================================================

Todos los features se calculan en el instante T, mirando como maximo
120 segundos hacia atras. NINGUN feature mira hacia adelante.

Nomenclatura:
  _Ns  = lookback de N segundos (ej: _5s = ultimos 5 segundos)
  spot = mercado spot
  fut  = mercado futuros
  L1, L5, L10, L20 = niveles del order book

Lookbacks estandar: [1s, 3s, 5s, 10s, 30s, 60s]
Lookbacks book:     [1s, 3s, 5s, 10s, 30s]

--------------------------------------------------------------------------------
GRUPO 1: ORDER BOOK SNAPSHOT (estado actual)
--------------------------------------------------------------------------------
Fuente: depth20 spot + futures (ultimo snapshot disponible)

Spot:
  spot_spread_bps               (best_ask - best_bid) / mid * 10000
  spot_microprice_dist          (microprice - mid) / mid * 10000
                                microprice = (bid*ask_qty + ask*bid_qty) / (bid_qty + ask_qty)
  spot_imbalance_L1             (bid_qty_1 - ask_qty_1) / (bid_qty_1 + ask_qty_1)
  spot_imbalance_L5             idem top 5 niveles
  spot_imbalance_L10            idem top 10 niveles
  spot_imbalance_L20            idem top 20 niveles
  spot_bid_depth_ratio          sum(bid_qty_20) / (sum(bid_qty_20) + sum(ask_qty_20))
  spot_depth_concentration_bid  qty_L1 / sum(qty_L20) del lado bid
  spot_depth_concentration_ask  qty_L1 / sum(qty_L20) del lado ask
  spot_wall_bid                 1 si algun bid_qty > mean + 3*std, 0 si no
  spot_wall_ask                 1 si algun ask_qty > mean + 3*std, 0 si no

Futures (mismos 11 features, prefijo "fut_"):
  fut_spread_bps
  fut_microprice_dist
  fut_imbalance_L1 ... L20
  fut_bid_depth_ratio
  fut_depth_concentration_bid/ask
  fut_wall_bid/ask

Cross (spot vs futures):
  basis_bps                     (fut_mid - spot_mid) / spot_mid * 10000
  spread_diff_bps               fut_spread_bps - spot_spread_bps
  imbalance_diff_L5             fut_imbalance_L5 - spot_imbalance_L5

Subtotal: 25 features

--------------------------------------------------------------------------------
GRUPO 2: ORDER BOOK DINAMICA (como cambio el book)
--------------------------------------------------------------------------------
Fuente: comparacion de snapshots depth20 a lo largo del tiempo

Para cada lookback [1s, 3s, 5s, 10s, 30s]:

  Spot:
    spot_imbalance_L5_delta_{N}s      imbalance_L5 ahora - imbalance_L5 hace N seg
    spot_bid_depth_ratio_delta_{N}s   bid_depth_ratio ahora - hace N seg
    spot_spread_delta_bps_{N}s        spread_bps ahora - hace N seg
    spot_microprice_move_{N}s         microprice ahora vs hace N seg (en bps)

  Futures (mismos 4, prefijo "fut_"):
    fut_imbalance_L5_delta_{N}s
    fut_bid_depth_ratio_delta_{N}s
    fut_spread_delta_bps_{N}s
    fut_microprice_move_{N}s

5 lookbacks x 4 features x 2 mercados = 40 features

Cross:
  basis_delta_{N}s              cambio en basis en ultimos N seg (5 valores)

Subtotal: 45 features

--------------------------------------------------------------------------------
GRUPO 3: TRADE FLOW (flujo de ordenes)
--------------------------------------------------------------------------------
Fuente: aggTrades spot + futures

Para cada lookback [1s, 3s, 5s, 10s, 30s, 60s]:

  Spot:
    spot_buy_pct_{N}s               buy_vol / total_vol (0 a 1)
    spot_signed_vol_zscore_{N}s     (buy_vol - sell_vol) normalizado:
                                    z-score contra media/std de ventana 60s
    spot_trade_intensity_{N}s       trade_count / N (trades por segundo)
                                    normalizado: ratio vs intensidad 60s
    spot_avg_size_ratio_{N}s        avg_trade_size / median_trade_size_60s
    spot_large_trade_imb_{N}s       (large_buys - large_sells) / total_large_trades
                                    large = qty > P95 de la ventana 60s

  Futures (mismos 5, prefijo "fut_"):
    fut_buy_pct_{N}s
    fut_signed_vol_zscore_{N}s
    fut_trade_intensity_{N}s
    fut_avg_size_ratio_{N}s
    fut_large_trade_imb_{N}s

6 lookbacks x 5 features x 2 mercados = 60 features

Cross:
  fut_spot_vol_ratio_{N}s       fut_volume / spot_volume para cada lookback
                                (quien lidera en actividad?)

6 lookbacks x 1 = 6 features

Subtotal: 66 features

--------------------------------------------------------------------------------
GRUPO 4: PRICE MOMENTUM
--------------------------------------------------------------------------------
Fuente: aggTrades (o bookTicker) spot

Para cada lookback [1s, 3s, 5s, 10s, 30s, 60s]:
  return_{N}s                   (price_now - price_Ns_ago) / price_Ns_ago
  realized_vol_{N}s             std(returns tick-a-tick) en esa ventana

Derivados:
  price_slope_10s               pendiente regresion lineal sobre returns ultimos 10s
  price_slope_30s               idem 30s
  price_acceleration            slope_10s - slope_30s
  return_skew_30s               skewness de returns en 30s
  return_skew_60s               skewness de returns en 60s
  max_drawdown_60s              maximo drawdown relativo en 60s
  max_runup_60s                 maximo runup relativo en 60s

Futures price vs spot:
  fut_return_{N}s               return del precio futuros (6 lookbacks)
  fut_leads_spot_{N}s           fut_return_{N}s - spot_return_{N}s
                                positivo = futuros subieron mas = futuros lideran

6 + 6 + 5 + 2 + 6 + 6 = 31 features

Subtotal: 31 features

--------------------------------------------------------------------------------
GRUPO 5: BOOK TICKER (best bid/ask en real-time)
--------------------------------------------------------------------------------
Fuente: bookTicker spot + futures

  spot_mid_vs_depth_mid         midprice de bookTicker vs midprice de depth20

  fut_mid_vs_depth_mid          idem futuros

  spot_bookticker_updates_{N}s  cantidad de updates en N seg (para N=1s, 5s, 10s)
  fut_bookticker_updates_{N}s   idem (actividad del top of book)

  spot_bid_ask_flip_rate_{N}s   veces que best_bid o best_ask cambio / N
  fut_bid_ask_flip_rate_{N}s    idem

2 + 6 + 6 = 14 features

GRUPO 6: DERIVADOS (contexto)

Funding y basis:

--------------------------------------------------------------------------------
GRUPO 7: VOLATILIDAD / REGIMEN
--------------------------------------------------------------------------------
Fuente: calculado de los datos de precio y volumen

================================================================================
6. ARQUITECTURA DEL SISTEMA
================================================================================

Tres componentes principales:

  A. Captura (24/7)
  B. Generacion de dataset + entrenamiento (offline)
  C. Prediccion en vivo (live)

--- A. CAPTURA ---

  Corre 24/7 en background.
  Recibe datos de los 10 streams y los guarda en parquet.
  Particionado por dia: data/raw/YYYY-MM-DD/<stream>/chunk_*.parquet

  Tambien se puede poblar con datos historicos de Binance Data Vision
  para tener un dataset inicial sin esperar semanas de captura.

  Almacenamiento estimado: ~2-4 GB/dia
  (el order book a 100ms es lo mas pesado)

--- B. GENERACION DE DATASET + ENTRENAMIENTO ---

  Ver seccion 6B para detalle completo del proceso de generacion.

--- C. PREDICCION EN VIVO ---

  1. Conectar a los 10 streams
  2. Mantener buffers circulares de 120 segundos para cada stream
  3. Cada 1 segundo:
     a. Calcular los ~213 features desde los buffers
     b. Pasar al modelo LightGBM (multiclass, 5 clases)
     c. Obtener 5 probabilidades
     d. Mostrar:
        T=16:35:42 | Price: 87,451.5 | SD:0.03 D:0.08 F:0.15 U:0.34 SU:0.40 | STRONG_UP
  4. Los buffers descartan datos de mas de 120 segundos

  Latencia objetivo: <50ms desde que llega el tick hasta prediccion.
  LightGBM con ~213 features: inferencia <1ms.

6B. GENERACION DE DATASET - DETALLE COMPLETO
================================================================================

El dataset de entrenamiento se genera OFFLINE a partir de los datos crudos
capturados. Es un proceso de tres fases: cargar datos crudos, iterar
segundo a segundo generando samples, y guardar el resultado.

--- FASE 1: DATOS CRUDOS (lo que esta en disco) ---

La captura guarda eventos tal cual llegan, sin procesar. Cada stream
tiene su propia carpeta con archivos parquet particionados por dia.

Ejemplo de lo que hay en disco para un dia:

--- FASE 2: GENERACION DE SAMPLES (1 por segundo) ---

El script build_dataset.py procesa los datos crudos y genera 1 training
sample por cada segundo. El proceso:

  ALGORITMO:

  1. Cargar todos los parquets de un dia en memoria
     (o procesarlo por chunks de 1 hora si no entra en RAM)

  2. Pre-indexar por timestamp para busqueda rapida:
     - trades: ya estan ordenados por timestamp_ms
     - orderbook: agrupar snapshots por timestamp_ms
     - bookticker: ordenado por timestamp_ms
     - etc.

  3. Iterar segundo a segundo:

     Para T = dia_start + 120s ... dia_end - 15s:
     (empezamos en +120s porque necesitamos lookback,
      terminamos en -15s porque necesitamos el target futuro)

       a) Obtener midprice(T):
          - Buscar el ultimo bookTicker con timestamp_ms <= T
          - midprice_T = (bid_price + ask_price) / 2
          - Si no hay datos en los ultimos 5 segundos → skip sample

       b) Obtener midprice(T+15s):
          - Buscar el ultimo bookTicker con timestamp_ms <= T+15000
          - midprice_T15 = (bid_price + ask_price) / 2
          - Si no hay datos en los ultimos 5 segundos → skip sample

       c) Calcular target (5 clases):
          - change_bps = (midprice_T15 - midprice_T) / midprice_T * 10000
          - Si change_bps < -T2             → target = 0 (STRONG_DOWN)
          - Si change_bps ∈ [-T2, -T1)      → target = 1 (DOWN)
          - Si change_bps ∈ [-T1, +T1]      → target = 2 (FLAT)
          - Si change_bps ∈ (+T1, +T2]      → target = 3 (UP)
          - Si change_bps > +T2             → target = 4 (STRONG_UP)

       d) Calcular features (llamar a feature_engine):
          - Filtrar trades spot donde timestamp_ms in [T-120s, T]
          - Filtrar trades futures idem
          - Buscar ultimo snapshot de orderbook spot con ts <= T
          - Buscar snapshot de orderbook spot de hace 1s, 3s, 5s, 10s, 30s
          - Idem para orderbook futures
          - Filtrar bookTicker spot/futures in [T-120s, T]
          - Buscar ultimo mark_price con ts <= T
          - Filtrar liquidaciones in [T-60s, T]
          - Buscar ultimo open_interest con ts <= T
          - Buscar ultimo long_short con ts <= T
          - Pasar todo al feature_engine → dict de ~213 features

       e) Guardar fila:
          {timestamp_ms: T, target: 0..4, change_bps: val, feat_1: val, ..., feat_213: val}

  EJEMPLO DE 4 SAMPLES CONSECUTIVOS:

--- OPTIMIZACIONES DE BUILD_DATASET ---

  Recalcular todo de cero para cada segundo seria extremadamente lento.
  Optimizaciones criticas:

  1. CARGAR DIA COMPLETO EN MEMORIA
     Cargar todos los parquets de un dia. El order book es lo mas pesado
     (~34M filas/dia spot + futures). Si no entra en RAM, procesar por
     chunks de 1 hora con overlap de 120s al inicio.

  2. PRE-INDEXAR POR TIMESTAMP
     Crear un indice (ej: sorted array de timestamps) para hacer busqueda
     binaria O(log n) en vez de filtrado lineal O(n).
     Para buscar "ultimo snapshot de book antes de T":
       idx = np.searchsorted(book_timestamps, T, side='right') - 1

  3. VENTANA DESLIZANTE
     Al avanzar de T a T+1s, la mayoria de los datos en la ventana de
     120s son los mismos. Solo cambia:
       - Se agrega ~1 segundo de datos nuevos por la derecha
       - Se quita ~1 segundo de datos viejos por la izquierda
     Para lookbacks cortos (1s, 3s), los datos cambian completamente.
     Para lookbacks largos (60s), cambia <2% de los datos.
     No se recalcula todo: se mantienen acumuladores incrementales donde
     sea posible (ej: conteo de trades, suma de volumen).

  4. PARALELIZAR POR DIA
     Cada dia es independiente (excepto los 120s de overlap al inicio).
     Se pueden procesar multiples dias en paralelo con multiprocessing.

  5. BATCH DE FEATURES
     En vez de calcular cada feature individualmente con filtrados de
     pandas, agrupar operaciones similares:
       - Todos los lookbacks de trades en un solo pass
       - Todos los lookbacks de order book en un solo pass

  TIEMPO ESTIMADO:
    Sin optimizar: ~30-60 min por dia
    Optimizado:    ~5-10 min por dia
    30 dias:       ~3-5 horas total

7. MODELO: LightGBM
================================================================================

--- POR QUE LIGHTGBM (y no XGBoost) ---

  Para este proyecto, LightGBM es mejor que XGBoost:

  1. TRAINING 5-10x MAS RAPIDO
     LightGBM usa histogram-based splitting y crece leaf-wise (en vez
     de level-wise como XGBoost). Con 2.6M+ samples, entrenar toma
     ~2-4 min en vez de ~15-20 min. Importa cuando re-entrenas seguido.

  2. MENOR USO DE RAM
     Histogram binning (256 bins por defecto) consume mucho menos
     memoria que los splits exactos de XGBoost. En un MacBook con
     datasets de millones de filas, esto importa.

  3. INFERENCIA MAS RAPIDA
     ~0.5-1ms por sample vs ~1-3ms de XGBoost. Para prediccion cada
     1 segundo esto da margen.

  4. MISMA ACCURACY
     Para datos tabulares numericos, ambos producen resultados
     practicamente identicos. No hay razon para elegir el mas lento.

  5. MANEJO NATIVO DE NaN
     LightGBM maneja NaN nativamente (usa el hijo con mayor ganancia).
     Critico porque nuestro dataset tiene NaN en features de streams
     secundarios (liquidaciones, OI, long/short).

--- HIPERPARAMETROS INICIALES ---

LightGBM:
  objective:          multiclass
  num_class:          5
  metric:             multi_logloss
  boosting_type:      gbdt
  num_leaves:         63        (equivalente a ~max_depth 6)
  learning_rate:      0.03
  n_estimators:       2000      (con early stopping)
  early_stopping_rounds: 100
  subsample:          0.8       (bagging_fraction)
  subsample_freq:     1         (bagging cada iteracion)
  colsample_bytree:   0.7       (feature_fraction)
  min_child_samples:  100       (alto: millones de samples, evita overfitting)
  reg_alpha:          0.1       (L1 regularization)
  reg_lambda:         1.0       (L2 regularization)
  max_bin:            255       (histogram bins, default)
  verbose:            -1
  seed:               42

Notas:
  - num_leaves=63 es el control principal de complejidad en LightGBM
    (en vez de max_depth). Leaf-wise growth con 63 hojas da arboles
    de profundidad ~6 pero mas flexibles.
  - min_child_samples=100 es alto porque con ~2.6M samples por mes,
    necesitamos hojas con suficiente evidencia estadistica.
  - learning_rate bajo + n_estimators alto + early stopping es mejor
    que learning_rate alto + pocos rounds.
  - colsample_bytree=0.7 para reducir correlacion entre arboles.

--- TRAINING CODE (pseudocodigo) ---

  import lightgbm as lgb

  dtrain = lgb.Dataset(X_train, label=y_train)
  dval   = lgb.Dataset(X_test, label=y_test, reference=dtrain)

  params = {
      "objective": "multiclass",
      "num_class": 5,
      "metric": "multi_logloss",
      "boosting_type": "gbdt",
      "num_leaves": 63,
      "learning_rate": 0.03,
      "subsample": 0.8,
      "subsample_freq": 1,
      "colsample_bytree": 0.7,
      "min_child_samples": 100,
      "reg_alpha": 0.1,
      "reg_lambda": 1.0,
      "verbose": -1,
      "seed": 42,
  }

  model = lgb.train(
      params,
      dtrain,
      num_boost_round=2000,
      valid_sets=[dval],
      callbacks=[
          lgb.early_stopping(100),
          lgb.log_evaluation(50),
      ],
  )

  # Guardar
  model.save_model("models/lightgbm_v1.txt")

  # Inferencia
  y_pred_proba = model.predict(X_test)  # → array de probabilidades

8. EVALUACION Y METRICAS
================================================================================

Metricas principales:
  - Accuracy global (5 clases)
  - Confusion matrix 5x5 (critico: ver donde se confunde)
  - F1 score macro y por clase
  - AUC-ROC one-vs-rest por clase (especialmente STRONG_UP y STRONG_DOWN)
  - Calibracion por clase: cuando dice P(STRONG_UP)=0.4, pasa ~40%?

Metricas por clase (lo que importa para operar):
  - Precision de STRONG_UP: cuando predice STRONG_UP, cuantas veces acierta?
  - Precision de STRONG_DOWN: idem
  - Recall de STRONG: cuantos STRONG reales detecta?
  - Tasa de confusion STRONG <-> opuesto (peor error: decir STRONG_UP y que
    sea STRONG_DOWN o viceversa)
  - % de samples donde clase predicha es FLAT (cobertura de "no operar")

Rentabilidad simulada:
  - Simular operaciones solo cuando P(STRONG_UP) o P(STRONG_DOWN) > threshold
  - profit/loss por operacion (usando change_bps real)
  - cantidad de operaciones por dia
  - drawdown maximo
  - Sharpe ratio
  - Comparar vs operar con UP/DOWN moderados

Validacion temporal:
  - NUNCA random split
  - Train en primeros 80% del tiempo, test en ultimo 20%
  - Tambien: walk-forward validation (entrenar con N dias, testear dia N+1,
    avanzar, repetir)

12. MANEJO DE DESCONEXIONES Y CALIDAD DE DATOS
================================================================================

Los websockets se desconectan. Binance hace mantenimiento. Internet falla.
Hay que detectar los huecos y decidir que hacer con los samples afectados.

--- ESCENARIOS DE DESCONEXION ---

  Caso 1: gap corto (<5 segundos)
    ... datos ... | gap 3s | ... datos ...
    No pasa nada. Los features de lookback 5s+ siguen teniendo datos
    suficientes. Los de 1s y 3s ven menos trades, pero es lo mismo que
    un momento de baja actividad. No requiere accion especial.

  Caso 2: gap medio (5 - 60 segundos)
    ... datos ... | gap 30s | ... datos ...
    Los lookbacks cortos vuelven a funcionar al reconectar. Pero los
    lookbacks largos (30s, 60s) ven una ventana con hueco: datos viejos
    pre-gap + datos nuevos post-gap, sin el medio. Los features de
    dinamica de book comparan "ahora vs hace 10s", pero si ese snapshot
    de hace 10s cae en el gap, sale NaN o un valor incorrecto.
    → SAMPLES CONTAMINADOS. Descartar.

  Caso 3: gap largo (>60 segundos)
    ... datos ... | gap 5 min | ... datos ...
    Toda la ventana de 120s esta vacia o parcialmente vacia.
    Features son basura completa.
    → DESCARTAR.

  Caso 4: un solo stream se cae (ej: orderbook, pero trades sigue)
    Depende de que stream:
    - Streams criticos (trades spot, trades futures, book spot, book futures):
      → DESCARTAR el sample si falta cualquiera de estos.
    - Streams secundarios (liquidaciones, mark_price, OI, long_short):
      → Se puede mantener el sample con esos features en NaN.
      LightGBM maneja NaN nativamente (los manda al hijo con mayor ganancia).

--- SOLUCION: HEARTBEATS + VALIDACION ---

  1. EN LA CAPTURA: Registro de heartbeats

     Cada stream registra un "heartbeat" cada vez que recibe datos.
     Se guarda en un archivo aparte:

       data/raw/YYYY-MM-DD/heartbeats.parquet
       ┌───────────────┬────────────────┬──────────┐
       │ timestamp_ms  │ stream         │ status   │
       ├───────────────┼────────────────┼──────────┤
       │ 1741700000000 │ trades_spot    │ ok       │
       │ 1741700000000 │ orderbook_spot │ ok       │
       │ 1741700000000 │ trades_futures │ ok       │
       │ 1741700001000 │ trades_spot    │ ok       │
       │ 1741700001000 │ orderbook_spot │ ok       │
       │ 1741700001000 │ trades_futures │ gap      │  ← no llego nada
       │ ...           │ ...            │ ...      │
       └───────────────┴────────────────┴──────────┘

     Tambien se puede derivar post-hoc: si entre dos eventos consecutivos
     de un stream hay >5 segundos de diferencia, hay un gap.

  2. EN BUILD_DATASET: Validacion de cobertura por sample

     Antes de calcular features para un segundo T, verificar:

     STREAMS CRITICOS (si falta alguno, sample invalido):
       - trades_spot:     ultimo trade < 5s ago?
       - trades_futures:  ultimo trade < 5s ago?
       - orderbook_spot:  ultimo snapshot < 2s ago?
       - orderbook_futures: ultimo snapshot < 2s ago?

     STREAMS SECUNDARIOS (si faltan, features = NaN pero sample valido):
       - bookticker_spot/futures: si no hay, usar midprice de depth20
       - mark_price: si no hay, funding_rate = NaN
       - liquidations: si no hay, liq_* = 0 (es normal que no haya)
       - open_interest: si no hay, oi_change = NaN
       - long_short: si no hay, ratios = NaN

     REGLA DE DESCARTE:

       gap_trades_spot    = T - ultimo_trade_spot_ms
       gap_trades_futures = T - ultimo_trade_futures_ms
       gap_book_spot      = T - ultimo_snapshot_book_spot_ms
       gap_book_futures    = T - ultimo_snapshot_book_futures_ms

       max_gap_critico = max(gap_trades_spot, gap_trades_futures,
                             gap_book_spot, gap_book_futures)

       Si max_gap_critico > 5000 ms (5 segundos):
         → SKIP este sample (no generar fila en el dataset)

       Si max_gap_critico > 2000 ms (2 segundos):
         → Generar sample pero marcarlo con data_quality = "degraded"

       Si max_gap_critico <= 2000 ms:
         → Sample normal, data_quality = "ok"

     La columna data_quality se guarda en el dataset. Al entrenar se
     puede filtrar: usar solo "ok", o incluir "degraded" tambien.

  3. EN LIVE: Misma logica

     El live_predictor aplica la misma validacion. Si detecta que un
     stream critico tiene gap > 5s:
       → No emite prediccion
       → Muestra: "T=16:35:42 | NO SIGNAL (gap en orderbook_futures: 8s)"

     Cuando el stream reconecta, espera a tener 5-10 segundos de datos
     frescos antes de volver a emitir predicciones (warm-up).

13. LIMITACIONES Y RIESGOS CONOCIDOS
================================================================================

  - Con pocos datos el modelo no aprende nada (accuracy ~50%).
    Minimo recomendado: 7 dias de datos. Ideal: 30+ dias.

  - Los datos historicos de Binance Data Vision NO tienen order book
    a nivel de depth20 (solo tienen bookDepth agregado por porcentaje).
    Los features de order book seran menos precisos cuando se entrena
    con datos historicos vs datos capturados en vivo.
    Alternativa: tardis.dev tiene order book de 25 niveles tick-by-tick,
    bookTicker, liquidaciones, OI cada 30s, y datos de otros exchanges
    (Coinbase, Bybit, OKX). Es la fuente mas completa para historicos.

  - El mercado cambia de regimen. Un modelo entrenado en un mercado
    bull puede no funcionar en un mercado bear. Re-entrenar regularmente.

  - 15 segundos es un horizonte MUY corto. El edge (si existe) sera
    pequeño. Necesitamos muchas predicciones con pequeño edge consistente,
    no pocas predicciones con gran edge.

  - Latencia de ejecucion: si el modelo dice UP pero tardamos 2 segundos
    en actuar, el precio ya se movio. La ejecucion debe ser rapida.

  - Overfitting: con ~213 features y millones de samples, LightGBM puede
    overfittear patrones espurios. Validacion temporal estricta y
    regularizacion agresiva son criticos.

  - Desconexiones: ver seccion 12 para el manejo completo. En promedio
    se espera perder 1-3% de los samples por gaps de conexion.
